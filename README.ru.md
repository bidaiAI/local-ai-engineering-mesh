# Local AI Engineering Mesh

[中文](README.zh-CN.md) | [English](README.md) | [Русский](README.ru.md) | [日本語](README.ja.md) | [Français](README.fr.md)

Превратите свои AI-инструменты из разрозненных помощников в инженерную систему с управлением, памятью и проверяемым завершением работы.

Это не набор промптов и не коллекция skills. Это репозиторий как operating model: способ поднять локальные AI-инструменты с уровня «умеет отвечать» до уровня «умеет работать по инженерной дисциплине».

Ключевая идея проста: Codex не должен работать в одиночку. Codex, Cursor, Antigravity, Claude Code и OpenCode могут быть специализированными узлами одной кроссплатформенной AI-системы под общей `AGENTS.md`.

## Какую проблему решает репозиторий

У большинства людей нет настоящей AI-системы. Есть только постоянное переключение между инструментами:
- один для кода
- один для исследований
- один для браузерных задач
- и запасной инструмент, когда основной кажется слабым

Это приводит к типичному набору проблем:
- постоянное переключение
- дублирование инструкций
- отсутствие общей памяти
- плавающий стиль и уровень качества
- зависимость от одного продукта, который «сейчас удобнее»

Этот репозиторий нужен, чтобы превратить набор локальных AI-инструментов в единую систему.

## Что здесь улучшается

- `shared law`: единый верхний слой правил через `AGENTS.md`
- `memory`: постоянные правила, маршрутизация и состояние проекта
- `bootstrap`: начальная инициализация команд, памяти и границ риска
- `orchestration`: явные роли для планирования, дизайна, проверки и релиза
- `planning`: сначала исследование, потом структурированный план исполнения
- `evidence`: завершение требует доказательств, а не только слов
- `release safety`: preview-first и publish gates
- `evolution`: повторяющиеся успешные паттерны становятся повторно используемыми навыками

## Снимок реальной системы

Этот репозиторий основан на реально работающей локальной конфигурации. На момент написания README в ней есть:
- `30` skills в `$CODEX_HOME/skills`
- `3` глобальных memory-файла в `$CODEX_HOME/memories`
- `10` governance-команд в `<project>/.codex/commands`
- `5` project memory-файлов в `<project>/.codex/memory`
- активные state-файлы в `<project>/.codex/state`

То есть это не теория, а уже работающая система на уровнях:
- rules
- skills
- project runtime
- state
- evidence

## Оценка четырёх инструментов на этой машине

Это не универсальные бенчмарки, а архитектурная оценка конкретной локальной конфигурации по состоянию на **25 марта 2026 года**.

### Codex — 92/100
**Роль:** главный центр инженерного исполнения и управления.

**Сильные стороны**
- самый сильный локальный исполнительный центр в этой системе
- лучшие governance skills, команды, release discipline и evidence workflow
- лучше всего подходит для превращения планов в дисциплинированную поставку результата

**Слабые стороны**
- по ощущениям от продукта и встроенной автоматизации иногда менее плавный, чем Claude

### Claude — 90/100
**Роль:** движок методологии workflow и эталонный стек.

**Сильные стороны**
- самый сильный command-and-workflow experience
- зрелая модульность мышления, привычки ревью и совместной работы
- остаётся важным эталоном в AI coding workflow

**Слабые стороны**
- в этой локальной системе не так глубоко встроен в execution governance, как Codex

### Cursor — 89/100
**Роль:** IDE-поле боя и editor-native слой реализации.

**Сильные стороны**
- ближе всех к самой поверхности кодинга
- высокая плотность project rules прямо в редакторе
- отлично снижает повседневное трение разработки

**Слабые стороны**
- слабее по system governance и operating-system feel, чем Codex и Antigravity

### Antigravity — 91/100
**Роль:** платформа расширения возможностей.

**Сильные стороны**
- широкий охват: browser, artifacts, knowledge, extended skills
- ощущается скорее как платформа, а не как один инструмент
- хорошо подходит для расширения возможностей и междоменной работы

**Слабые стороны**
- из-за широты может добавлять шум
- на основном инженерном пути не всегда так же стабилен, как Codex

### Итоговая локальная система — 91/100
Самая сильная сторона этой машины не в том, что один AI лучший. Сильна именно организация системы:
- единый operating law
- несколько специализированных endpoint’ов
- project memory
- execution gates
- кроссплатформенная согласованность

Самое точное определение:

**Shared Law + Multi-Tool Specialized AI System**

Это уже не настройка в стиле «выбери один инструмент».
Это coordinated local AI engineering mesh.

## Кроссплатформенная структура

В реальной среде за этим репозиторием:
- `$CODEX_HOME/config.toml` загружает `$SHARED_LAW_HOME/AGENTS.md` через `shared-law loader`
- `$CODEX_HOME/instructions.md` отвечает за Codex-specific routing, profiles и guidance по skills
- несколько coding tools используют один верхнеуровневый operating law

Это означает:
- нативность на уровне исполнения
- кроссплатформенную согласованность на уровне политики

И значит, что при смене инструмента workflow не надо перезапускать с нуля.

См. [CROSS-PLATFORM.md](docs/CROSS-PLATFORM.md).

## Сравнение с актуальным Claude

По состоянию на **23 марта 2026 года** публичный coding stack Claude у Anthropic возглавляют:
- `Claude Opus 4.6` — опубликован **5 февраля 2026 года**
- `Claude Sonnet 4.6` — опубликован **17 февраля 2026 года**

Этот репозиторий **не** утверждает, что локальный инструмент автоматически сильнее Claude по raw model capability.

Здесь делается более узкое, но практичное утверждение:

Если усилить operating layer локальных AI-инструментов, они становятся гораздо конкурентоспособнее в длинном инженерном цикле, потому что workflow перестаёт зависеть от одного идеального промпта или одного всегда доступного продукта.

См. [COMPARE-WITH-CLAUDE.md](docs/COMPARE-WITH-CLAUDE.md).
